DESIGN DECISIONS - TinyLlama U55C Inference Accelerator
=======================================================
Last updated: 2026-04-13
Author: Samarth (Dataflow, Tiling, Control)

This file tracks the current architecture decisions for the post-Phase-1
TinyLlama implementation target on the Xilinx Alveo U55C. The goal is no
longer a single-token toy pipeline. The goal is multi-token prompt ingestion
plus autoregressive token generation for TinyLlama 1.1B using a mixed-precision
FPGA accelerator.

---

SECTION A - CURRENT ARCHITECTURE DECISIONS
==========================================

A1. Target model and inference scope
  Model: TinyLlama/TinyLlama-1.1B-Chat-v1.0
  hidden_size=2048, intermediate_size=5632, num_hidden_layers=22,
  num_attention_heads=32, num_key_value_heads=4, head_dim=64,
  vocab_size=32000, max_position_embeddings=2048, rope_theta=10000.0.
  Goal: accept a prompt token sequence, perform prefill, then decode and emit
  generated token ids until EOS or max_new_tokens.

A2. Two runtime modes: prefill and decode
  Prefill consumes the full prompt sequence length S (1..2048), applies the
  causal mask across the prompt, and writes K/V for every processed position
  into the KV cache.
  Decode consumes one new token at a time, reads the existing KV cache,
  appends one new K row and one new V row per layer, and emits one next-token id.
  This removes the old seq_len=1-only simplification from the architecture.

A3. Control ownership: FPGA owns model execution, host owns software I/O
  Host responsibilities: tokenization/detokenization, model launch via XRT,
  prompt token upload, generation parameter upload, and generated token readback.
  FPGA responsibilities: embedding lookup, prefill scheduling, 22-layer execution,
  KV-cache management, final RMSNorm, LM head projection, greedy argmax, and
  the decode-step loop.
  The runtime interface is host-launch / FPGA-execute / host-readback.
  Step-by-step host control exists only in simulation/testbench infrastructure,
  not in the deployed FPGA runtime path.

A4. One decoder-layer engine reused 22 times
  TinyLlama has 22 decoder layers with the same block structure and tensor sizes.
  All 22 layers use the same datapath shape:
    RMSNorm -> Q/K/V -> RoPE -> attention scores -> softmax -> weighted sum ->
    O projection -> residual add -> RMSNorm -> gate/up -> SiLU*multiply ->
    down projection -> residual add
  The layer engine is instantiated once and reused for layer_idx = 0..21.
  What changes per layer: weights, two RMSNorm gamma vectors, and that layer's
  KV-cache region.
  What stays the same per layer: hidden size, FFN size, head counts, head_dim,
  and control/dataflow shape.

A5. GEMM engine reuse inside each layer
  The shared GEMM engine is reused for all GEMM-heavy operations:
    - Q projection
    - K projection
    - V projection
    - attention score computation
    - weighted sum with V
    - O projection
    - gate projection
    - up projection
    - down projection
    - LM head projection (tiled path)
  This means there are two levels of reuse:
    (i) one GEMM engine reused across all GEMMs in a layer
    (ii) one decoder-layer engine reused across all 22 layers

A6. Full TinyLlama attention architecture: 32 Q heads / 4 KV heads (GQA)
  The single-head Phase 1 simplification is removed.
  The accelerator implements grouped-query attention:
    - 32 query heads
    - 4 key/value heads
    - 8 query groups per KV head
  K/V repetition is logical reuse or address remapping, not a physical copy.
  RoPE is applied to Q and K after projection and before score computation.

A7. KV cache is required
  Each layer owns one K cache and one V cache.
  Prefill writes the prompt's K/V rows into cache.
  The cached K row format is post-RoPE K, not pre-RoPE K.
  The cached V row format is post-projection V.
  During decode, the new K row is RoPE-transformed before cache append, and
  cached K rows are read back directly without reapplying RoPE.
  Decode reads all previous cache rows and appends one new row per step.
  Storage location: HBM.
  Architectural format: symmetric INT8 cache with explicit scale metadata.
  K and V use separate scales.
  Scale granularity: per-layer, per-KV-head, separately for K and V.
  Cache layout in memory: [layer][kv_head][token][head_dim].
  The implementation never depends on "softmax = 1 because seq_len=1".

A8. Sequence support and masking
  Supported prompt length: 1..2048 tokens.
  Supported generation: until EOS or max_new_tokens.
  Prefill uses a true causal mask over the prompt.
  Decode uses query_len=1 and key_len=cache_len+1.
  Softmax is therefore a real sequence-length-dependent block, not a permanent
  passthrough stage.
  Because the production datapath is 512 lanes wide, the score-mask path is
  chunked as 8 query rows x 64 key columns per cycle.
  The fixed score chunk constants are:
    - SCORE_ROWS_PER_CHUNK = 8
    - SCORE_K_TILE = 64
    - SCORE_CHUNKS_PER_TILE = 2
  Score-chunk lane packing is row-major:
    lane = query_row_local * 64 + key_col_local
  For score chunks, tag.elem_count represents query_row_count * 64.
  If key_col_count < 64 at a tail chunk, the extra columns are not represented
  via elem_count; they are explicitly forced to MASK_NEG_INF by the causal-mask
  unit before the softmax path.

A9. Mixed-precision / quantization split
  Quantized GEMM-heavy blocks:
    - embedding output path into the layer datapath
    - Q / K / V / O projections
    - attention score computation
    - weighted sum with V
    - gate / up / down projections
    - LM head projection
  Arithmetic for quantized GEMMs:
    - inputs: INT8
    - weights: INT8
    - accumulation: INT32
  Higher-precision blocks kept out of pure INT8:
    - RMSNorm
    - RoPE
    - softmax
    - final RMSNorm
  Residual accumulation remains INT32 before requantization.

A10. Quantization scale policy
  Quantization is symmetric INT8 with zero_point = 0 everywhere.
  Weight quantization: static per-output-channel scales.
  Activation quantization: static per-tensor scales per block output.
  RTL-visible scale payload format: unsigned Q16.16 multipliers.
  Requantization scale distribution: one scale entry per 32-lane bank slice.
  A full 512-lane activation tile therefore consumes 16 scale entries.
  KV-cache quantization: static per-layer, per-KV-head scales, stored separately
  for K and V.
  Calibration is performed offline from the TinyLlama reference path.
  RoPE preserves activation magnitude scale. The FPGA therefore uses the same
  Q or K activation scale before and after the RoPE stage; no extra RoPE-only
  scale metadata path exists in the production datapath.

A11. Final RMSNorm, LM head, and token selection are on FPGA
  After layer 21 output, the FPGA applies final RMSNorm, performs the LM head
  projection as a tiled GEMM, and computes greedy argmax.
  The FPGA emits token ids; the host detokenizes them for display.
  Top-k and top-p sampling are not part of the initial implementation.

A12. Memory residency on U55C
  All model weights reside in HBM, including embeddings and LM head weights.
  KV cache resides in HBM.
  On-chip BRAM/URAM stores:
    - activation tiles
    - current Q/K/V/O tiles
    - partial sums / accumulator tiles
    - scale registers and control metadata
  Full-model on-chip residency is not required.
  The design uses the U55C HBM subsystem as the model-state store.
  Board memory organization: 16 GB HBM exposed as 32 pseudo-channels
  of 512 MB each.

A13. Weight streaming and buffering
  Double buffering / ping-pong buffering is mandatory for the full design, not
  an optional later optimization.
  The accelerator overlaps:
    - HBM weight tile fetch
    - HBM KV-cache fetch/write
    - on-chip compute
  Separate buffer sets are used for weights, activations, and KV tiles to avoid
  stalling the GEMM engine.

A14. On-chip controller replaces per-GEMM host sequencing
  The controller sequence is:
    prompt tokens -> embedding lookup -> prefill -> decode loop ->
    final token selection -> stop on EOS/max_new_tokens
  Inside each decode step, the controller iterates:
    layer_idx = 0..21
  and inside each layer it schedules the required block sequence and memory moves.
  The host never issues a PCIe round trip per GEMM in the deployed design.

A15. Throughput target and engine width
  The implementation target is a 512-lane INT8 GEMM engine on U55C.
  The existing 8-lane path remains simulation and validation infrastructure only.
  The production accelerator uses the same control/dataflow pattern with the
  512-lane engine width.

A16. Accuracy bring-up path
  Validation order:
    1. floating-point TinyLlama reference
    2. mixed-precision software bridge
    3. one quantized decoder layer on FPGA
    4. 22-layer quantized decode step on FPGA
    5. full prefill + decode + token generation
  The implementation includes a debug mode that dumps per-layer activations and
  KV-cache slices for comparison against the software reference.

SECTION B - FINALIZED IMPLEMENTATION CHOICES
===========================================

B1. LM head precision policy
  The LM head is implemented on FPGA as a tiled symmetric-INT8 GEMM:
    - activations: INT8
    - weights: INT8
    - accumulation: INT32
    - weight scales: static per-output-channel
  Greedy argmax is computed after the FPGA-side LM head.
  There is no higher-precision LM-head path in the production bitstream.
  A higher-precision LM-head path exists only in a separate validation build.

B2. KV-cache precision policy
  The production KV cache is symmetric INT8 in HBM.
  K and V are stored separately.
  Cached K rows are stored after RoPE has already been applied.
  Cached V rows are stored after the V projection output is produced.
  Scale metadata is stored separately for K and V.
  Scale granularity is fixed to per-layer, per-KV-head.
  Cache layout is fixed to:
    [layer][kv_head][token][head_dim]
  Prefill writes one K row and one V row for every token position.
  Decode reads the full cached prefix and appends one new K row and one new
  V row per layer.
  The production bitstream does not include an FP16 or FP32 KV-cache mode.
  Any higher-precision cache path exists only in software or debug validation.

B3. Prefill tiling policy
  Prefill is sequence-tiled with a fixed tile length of 64 tokens.
  Prompt tokens are processed tile by tile until the full prompt is consumed.
  Attention-score and softmax working sets are computed one sequence tile at a time.
  HBM layout for prompt-dependent tensors is fixed as:
    - activations: [token][feature]
    - KV cache: [layer][kv_head][token][head_dim]
  HBM layout for weight tensors is fixed as:
    [layer][tensor][output_channel][input_channel_tile]
  with output channels contiguous to match per-output-channel quantization.

B4. Sampling feature set
  The initial implementation performs greedy argmax only.
  Top-k and top-p are excluded from the first hardware implementation.
  The token emitted by the FPGA is always the argmax token id from the final logits.

B5. HLS vs RTL partition for non-linear blocks
  SystemVerilog RTL implements:
    - prefill/decode controller
    - GEMM engine
    - requantization
    - residual add
    - embedding address generation
    - KV-cache address/control logic
    - HBM read/write scheduling
    - RoPE datapath using precomputed sin/cos tables
    - LM-head tiling
    - greedy argmax
  HLS C++ implements:
    - RMSNorm
    - softmax
    - SiLU
  HLS math uses fixed-point approximations.
  Vendor floating-point or math IP is not part of the initial implementation.
  RoPE is implemented in RTL over fixed 512-lane head slices of:
    - ROPE_CHUNK_TOKENS = 8 tokens
    - HEAD_DIM = 64 dimensions
  RoPE ROM tables are precomputed offline into Q16.16 sine/cosine memories and
  read by the RTL rotary datapath. The ROM contents are generated from the
  TinyLlama reference/export path and consumed directly by synthesis/simulation.

B6. HBM channel allocation
  The U55C memory plan uses 32 HBM pseudo-channels, PC00-PC31.
  Allocation is fixed as:
    - PC00-PC15: decoder-layer weights
    - PC16-PC17: token embeddings, final RMSNorm gamma, quantization metadata
    - PC18-PC21: LM-head weights
    - PC22-PC25: K cache
    - PC26-PC29: V cache
    - PC30: generated-token ring buffer, host command block, status block
    - PC31: debug capture buffer
  Decoder-layer weights are striped across PC00-PC15 by output-channel tile.
  LM-head weights are striped across PC18-PC21 by vocab tile.
  K and V caches are striped across their assigned PCs by layer and KV head.

B7. Stop-condition ownership
  The FPGA owns the decode loop stop conditions.
  The FPGA stops generation on the first occurrence of either:
    - emitted token == EOS
    - generated token count == max_new_tokens
  The host launches generation, polls status, and reads completed token ids.
  The host does not drive one token at a time in the production runtime path.

B8. Debug / observability contract
  Debug capture is optional and disabled in performance builds.
  When enabled, debug data is written into the dedicated HBM debug buffer on PC31.
  The exported tensors are fixed to:
    - layer input activation slice
    - quantized Q slice
    - quantized K slice
    - quantized V slice
    - attention-score window
    - post-softmax window
    - updated K-cache row
    - updated V-cache row
    - layer output activation slice
    - final hidden-state slice
    - logits top-16 slice
    - chosen token id
    - quantization scales used by the captured block
  Captures are layer-scoped or decode-step-scoped, not full unbounded traces.
  Debug data is read back by the host after the selected step completes.

B9. AXI-Lite control-register map and launch semantics
  The control-plane register interface is fixed to 32-bit AXI-Lite words.
  All 64-bit base addresses are exposed as low/high 32-bit register pairs.
  The register map is fixed as:
    - word 0: CONTROL
    - word 1: STATUS
    - word 2: CMD_BASE_LO
    - word 3: CMD_BASE_HI
    - word 4: STATUS_BASE_LO
    - word 5: STATUS_BASE_HI
    - word 6: DEBUG_BASE_LO
    - word 7: DEBUG_BASE_HI
    - word 8: PROMPT_TOKEN_COUNT
    - word 9: MAX_NEW_TOKENS
    - word 10: EOS_TOKEN_ID
    - word 11: DEBUG_CFG
    - word 12: GENERATED_TOKEN_COUNT
    - word 13: LAST_TOKEN_ID
    - word 14: CURRENT_LAYER
    - word 15: CURRENT_BLOCK
    - word 16: RTL_VERSION
  CONTROL bit definitions are fixed as:
    - bit 0: START (write-one pulse)
    - bit 1: MODE (0 = prefill launch, 1 = decode-only launch)
    - bit 2: ABORT_REQ (sticky until a new launch or hardware completion clears it)
  STATUS bit definitions are fixed as:
    - bit 0: BUSY
    - bit 1: DONE (sticky)
    - bit 2: ERROR (sticky)
    - bit 3: STOP_VALID (sticky)
    - bits [6:4]: STOP_REASON
    - bits [11:8]: ERROR_CODE
  DEBUG_CFG bit definitions are fixed as:
    - bit 0: DEBUG_ENABLE
    - bits [8:4]: DEBUG_LAYER_SEL
    - bits [19:12]: DEBUG_STEP_SEL
  Launch semantics are fixed as:
    - MODE = 0 and MAX_NEW_TOKENS = 0 -> prefill-only execution
    - MODE = 0 and MAX_NEW_TOKENS > 0 -> prefill followed by decode loop
    - MODE = 1 -> decode-only execution using a pre-existing KV cache
  The first emitted token after a normal prompt launch is produced immediately
  after the prefill pass completes and the FPGA executes final RMSNorm, LM head,
  and greedy argmax for the prompt tail position.

B10. PC30 host command/status block layout
  The command block and status block written through PC30 are fixed to one
  256-bit beat each.
  Command block word layout is fixed as:
    - word 0: PROMPT_TOKENS_BASE_LO
    - word 1: PROMPT_TOKENS_BASE_HI
    - word 2: GENERATED_TOKENS_BASE_LO
    - word 3: GENERATED_TOKENS_BASE_HI
    - word 4: GENERATED_TOKENS_CAPACITY
    - words 5..7: reserved
  Status block word layout is fixed as:
    - word 0: STATUS bits using the same bit packing as the AXI-Lite STATUS register
    - word 1: GENERATED_TOKEN_COUNT
    - word 2: LAST_TOKEN_ID
    - word 3: CURRENT_LAYER
    - word 4: CURRENT_BLOCK
    - words 5..6: reserved
    - word 7: RTL_VERSION
  `host_cmd_status_mgr.sv` reads the command beat from `cmd_base_addr` on every
  launch, exposes the parsed prompt/ring-buffer addresses to downstream blocks,
  and writes one status beat to `status_base_addr` when terminal status changes
  occur.
  The host software and FPGA runtime must both follow this exact layout.

B11. Internal DMA descriptor contract
  All internal HBM-facing DMA paths use a split descriptor/data handshake:
    - read path: descriptor valid/ready, then beat data valid/ready
    - write path: descriptor valid/ready and write-data valid/ready
  The shared descriptor format is fixed to:
    - region
    - tensor_id
    - write_not_read
    - pseudo_channel
    - addr
    - burst_len (16 bits)
    - byte_count (32 bits)
    - layer_id
    - kv_head_id
    - tile_id
  `byte_count` is fixed to 32 bits because full KV-cache transfers can exceed
  65535 bytes.

B12. Verification policy and golden-trace usage
  Directed smoke tests remain mandatory for control, routing, DMA, and storage
  modules.
  Golden traces are not required for Phase 0, Phase 1, or Phase 2.
  Golden traces are recommended for Phase 3 arithmetic blocks.
  Golden traces are required from Phase 4 onward for math/dataflow blocks and
  are required for Phase 7-9 integration gates.
  The golden-trace export and consumption rules are defined in:
    docs/golden_trace_plan.md
  The canonical export root is fixed to:
    sim/golden_traces/

SECTION C - REUSABLE BUILDING BLOCKS ALREADY IMPLEMENTED
========================================================

C1. Output-stationary accumulation
  Each MAC lane holds its accumulator across the full K loop.
  Status: implemented in control_fsm.sv + mac_array.sv for the validation core.

C2. Tile size T = 8 validation configuration
  One tile = 8 output elements in the simulation path.
  Status: implemented and verified for the validation core.

C3. FSM state pattern IDLE -> LOAD -> COMPUTE -> WRITE -> (LOAD | DONE)
  LOAD absorbs BRAM latency; mac_valid stays low outside COMPUTE.
  Status: implemented in the current validation FSM.

C4. 8-lane INT8 MAC array
  INT8 x INT8 -> INT32, 8 lanes in parallel.
  Status: implemented and verified against test vectors.

C5. Behavioral BRAM stubs in top.sv
  Used for simulation and early integration.
  Status: implemented.

C6. Host-side preload/write ports
  Current top.sv exposes write ports for x and w buffers for testbench / XRT use.
  Status: implemented as reusable infrastructure.

C7. Offline quantization and test-vector generation
  model.py and gen_test_vectors.py compute static scales and simulation vectors.
  Status: implemented and still relevant to the full design.

C8. Phase 1 artifacts remain validation infrastructure
  The old single-token, reduced-dimension flow remains useful for unit testing and
  simulation speed, but it is no longer the architecture definition for the final design.
